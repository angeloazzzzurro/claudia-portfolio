Il progetto

EthnicalBox è un'app editoriale per esplorare cucine etniche di Milano: categorie, ristoranti, piatti signature. L'obiettivo era costruire un'esperienza visiva ricca — stile guida food — con React e Vite, senza backend, usando dati JSON statici come fonte di verità.

La struttura segue il pattern Container/Presentational: i componenti smart gestiscono i dati e il routing, i componenti presentazionali ricevono solo props e non sanno da dove vengono i dati. Questo rende ogni UI testabile e riusabile indipendentemente.

Codice

Il componente RestaurantCard è puramente presentazionale — riceve props e renderizza, senza logica interna. Il click handler viene passato dal parent, così la card non sa mai cosa succede quando viene cliccata.

components/RestaurantCard.jsx
// Componente presentazionale — solo props, nessun side effect
export function RestaurantCard({ restaurant, onSelect }) {
  const { name, cuisine, rating, image, distance } = restaurant;

  return (
    <article
      className={styles.card}
      onClick={() => onSelect(restaurant)}
      role="button"
      tabIndex={0}
      onKeyDown={e => e.key === 'Enter' && onSelect(restaurant)}
      aria-label={`${name} — cucina ${cuisine}`}
    >
      <div className={styles.image}>
        <img src={image} alt="" aria-hidden="true" loading="lazy" />
        <span className={styles.badge}>{cuisine}</span>
      </div>
      <div className={styles.body}>
        <h3 className={styles.name}>{name}</h3>
        <div className={styles.meta}>
          <StarRating value={rating} />
          <span>{distance} km</span>
        </div>
      </div>
    </article>
  );
}
Container vs Presentational: RestaurantCard non chiama mai fetch, non usa useState, non conosce il routing. Questo la rende riutilizzabile ovunque nell'app e testabile con un semplice render con props mock.

Il custom hook useRestaurants incapsula il fetching e il filtering. Se in futuro i dati vengono da un'API reale, si cambia solo il hook — nessun componente UI va toccato.

hooks/useRestaurants.js
export function useRestaurants(categoryId = null) {
  const [restaurants, setRestaurants] = useState([]);
  const [loading, setLoading]         = useState(true);
  const [error, setError]             = useState(null);

  useEffect(() => {
    setLoading(true);
    setError(null);

    fetch('/data/restaurants.json')
      .then(r => {
        if (!r.ok) throw new Error(`HTTP ${r.status}`);
        return r.json();
      })
      .then(data => {
        const filtered = categoryId
          ? data.filter(r => r.categoryId === categoryId)
          : data;
        setRestaurants(filtered);
      })
      .catch(err => setError(err.message))
      .finally(() => setLoading(false));

  }, [categoryId]);   // re-fetch solo se cambia la categoria

  return { restaurants, loading, error };
}
Dependency array [categoryId]: useEffect si riattiva solo quando cambia la categoria — non ad ogni render. Senza questo, il fetch partirebbe ad ogni click, scroll, o aggiornamento di stato non correlato.

Il routing annidato di React Router v6 mappa perfettamente la gerarchia dei dati: categoria → ristorante → piatto.

App.jsx — routing structure
import { BrowserRouter, Routes, Route } from 'react-router-dom';

export default function App() {
  return (
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<HomePage />} />

        {/* Layout condiviso per tutte le pagine categoria */}
        <Route path="/category/:id" element={<CategoryLayout />}>
          <Route index element={<RestaurantList />} />
          <Route path="restaurant/:rid" element={<RestaurantDetail />} />
          <Route path="restaurant/:rid/dish/:did" element={<DishDetail />} />
        </Route>

        <Route path="*" element={<NotFound />} />
      </Routes>
    </BrowserRouter>
  );
}
Nested routes + Outlet: CategoryLayout renderizza la navbar della categoria e un <Outlet /> dove vengono iniettate le route figlie. La breadcrumb e il filtro categoria rimangono visibili mentre si naviga nei ristoranti.
