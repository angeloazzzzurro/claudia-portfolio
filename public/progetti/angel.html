Il progetto

Angel è un agente AI autonomo progettato per eseguire task complessi in più step senza supervisione continua. Si basa sul pattern ReAct (Reasoning + Acting): ad ogni iterazione il modello decide se rispondere direttamente o usare un tool, esegue l'azione, osserva il risultato e ragiona sul passo successivo.

L'obiettivo era costruire qualcosa di leggero ma reale — non un toy project ma un framework usabile per automazione quotidiana: ricerca web, manipolazione file, esecuzione codice in sandbox. La coda SQLite con WAL mode garantisce accesso concorrente da più processi (bot Telegram + REST API) senza lock contention.

Codice

Il loop principale dell'agente segue il pattern ReAct: il modello ragiona (Think), sceglie un'azione (Act), legge il risultato (Observe) e ripete. Il loop termina quando il modello risponde senza chiamare tool, o quando si raggiunge il numero massimo di step.

agent.py
async def run(self, task: str) -> str:
    self.memory.add("user", task)

    for step in range(self.max_steps):
        # Think: chiedi al modello cosa fare
        response = await self.llm.complete(
            messages=self.memory.get_context(),
            tools=self.tools.schemas(),
        )

        # Se il modello risponde senza tool → task completato
        if response.stop_reason == "end_turn":
            return response.content[0].text

        # Act: esegui il tool scelto dal modello
        tool_call = response.content[0]          # ToolUseBlock
        result = await self.tools.execute(
            name=tool_call.name,
            input=tool_call.input,
        )

        # Observe: aggiungi risultato al contesto
        self.memory.add_tool_result(tool_call.id, result)

    return f"[Agent] Max steps ({self.max_steps}) reached."
Perché un loop for e non while? Il limite di step è un safety net obbligatorio — un agente in loop infinito con accesso a shell e filesystem è un rischio reale. Il numero massimo è configurabile ma sempre presente.

Il tool dispatch usa il pattern Strategy: ogni tool implementa la stessa interfaccia, l'agente non sa come funzionano internamente. Aggiungere un nuovo tool è una sola classe con due metodi.

tools/base.py + tools/shell.py
# Interfaccia comune per tutti i tool
class BaseTool(ABC):
    name: str
    description: str

    @abstractmethod
    def schema(self) -> dict:
        """Schema JSON per l'Anthropic API."""
        ...

    @abstractmethod
    async def execute(self, **kwargs) -> str:
        """Esegue il tool e restituisce stringa al modello."""
        ...


# Implementazione: esecuzione shell sandboxed
class ShellTool(BaseTool):
    name = "shell"
    description = "Esegue un comando bash. Output max 4000 chars."

    def schema(self) -> dict:
        return {
            "name": self.name,
            "description": self.description,
            "input_schema": {
                "type": "object",
                "properties": {
                    "command": {"type": "string", "description": "Comando da eseguire"}
                },
                "required": ["command"],
            },
        }

    async def execute(self, command: str) -> str:
        proc = await asyncio.create_subprocess_shell(
            command,
            stdout=asyncio.subprocess.PIPE,
            stderr=asyncio.subprocess.STDOUT,
            timeout=30,                          # hard timeout
        )
        stdout, _ = await proc.communicate()
        return stdout.decode()[:4000]            # tronca output lungo
Strategy pattern: ToolRegistry carica tutti i tool all'avvio e li espone come lista di schema JSON all'API Anthropic. Il modello sceglie quale chiamare — il codice Python non decide mai quale tool usare.

La coda SQLite con WAL mode permette a bot Telegram, REST API e TUI di operare contemporaneamente sullo stesso database senza bloccarsi a vicenda.

queue_backend.py
class TaskQueue:
    def __init__(self, db_path: str):
        self.conn = sqlite3.connect(db_path, check_same_thread=False)
        # WAL: i reader non bloccano i writer e viceversa
        self.conn.execute("PRAGMA journal_mode=WAL")
        self.conn.execute("""
            CREATE TABLE IF NOT EXISTS tasks (
                id       TEXT PRIMARY KEY,
                payload  TEXT NOT NULL,
                status   TEXT DEFAULT 'pending',
                result   TEXT,
                created  REAL DEFAULT (unixepoch('now', 'subsec'))
            )
        """)
        self.conn.commit()

    def push(self, task_id: str, payload: dict) -> None:
        self.conn.execute(
            "INSERT INTO tasks (id, payload) VALUES (?, ?)",
            (task_id, json.dumps(payload))
        )
        self.conn.commit()

    def claim_next(self) -> dict | None:
        """Atomicamente prende e blocca il prossimo task pending."""
        row = self.conn.execute(
            "UPDATE tasks SET status='running' WHERE id = "
            "(SELECT id FROM tasks WHERE status='pending' "
            " ORDER BY created LIMIT 1) RETURNING *"
        ).fetchone()
        self.conn.commit()
        return dict(zip([c[0] for c in self.conn.description], row)) if row else None
UPDATE … RETURNING: L'operazione di claim è atomica — un solo processo può prendere un task anche se Telegram, REST API e TUI interrogano la coda contemporaneamente. Nessun lock esplicito necessario grazie a WAL.
